Recommended use of ADMS Identifier

Motivation

In the past ADMS provided in the context of the DCAT embedding a recommended use fo the Identifier class according to the UN/CEFACT specification. In the ADMS vocabulary release 2.00 this recommended use was made implicit.

In the context of the Core Business Vocabulary the need for documenting more formally this recommended use was identified. Not only to address the existing recommendation, but also to address additional needs. The intend of this section is not to create a complete ADMS application profile, but only to document the aspect of idenfitiers.

The ADMS Identifier class is used when identifiers cannot be treated as simple literal values, in which case dct:identifier is recommended to be used, but when the origin of the identifier is important or when one like to relate the identifier to the rules according to which the identifier has been constructed or assigned. It thus addresses the more complex identifier use cases.

The raised use case within the Core Business Vocabulary expressed the need for relating the identifier with its identifier schema. And documenting who issued the identifier and who governed the identifier schema. The data specification in this section addresses these needs. Users of the ADMS Identifiers are recommended to apply this data model in their specifications.

Overview

This section describes the usage of the following entities for a recommended usage of the Identifier class:
| Agent | Identifier | Identifier Schema |

And supported by these datatypes:
| Date | Literal | Text |

Agent

Definition
Entity that is able to carry out action.
Usage Note
In compliance with the description from FOAF, an Agent is considered as any entity that is able to carry out actions. The Agent class acts as a generic element which can be further specified by implementers for their usages, for example by defining the Person class from the Core Person Vocabulary or Organization from W3C's Organization Ontology as a subclasses of Agent. This Person or Organization can then issue a certain Requirement or be concerned by an Evidence provided.
Properties
For this entity the following properties are defined: issued by .
Property Range Card Definition Usage
issued by Identifier 0..* Agent legally responsible for the Identifier.

Identifier

Definition
A structured reference that identifies an entity.
Usage Note
The Identifier class is based on the UN/CEFACT class of the same name and is defined under the ADMS namespace.
Properties
For this entity the following properties are defined: date issued , notation , schema , schema agency .
Property Range Card Definition Usage
date issued Date 0..* Date of formal issuance of the identifier.
notation Literal 0..* A string of characters to uniquely identify an identifier.
schema Identifier Schema 0..* The schema used to construct the identifier.
schema agency Literal 0..* The name of the agency that issued the identifier. use dcterms:creator to link to a class describing the agency that manages the identifier scheme or adms:schemaAgency to provide the name as a literal.

Identifier Schema

Definition
The schema used to assign and manage an identifier to an entity.
Properties
For this entity the following properties are defined: schema name .
Property Range Card Definition Usage
schema name Text 0..* Name of the schema used to construct the identifier.

Datatypes

The following datatypes are used within this specification.
Class Definition
Date
Date represents top-open intervals of exactly one day in length on the timelines of dateTime, beginning on the beginning moment of each day, up to but not including the beginning moment of the next day). For non-timezoned values, the top-open intervals disjointly cover the non-timezoned timeline, one per day. For timezoned values, the intervals begin at every minute and therefore overlap.
Literal
The class of literal values, eg. textual strings and integers.
Text
The text data type is a combination of a string and a language identifier.
